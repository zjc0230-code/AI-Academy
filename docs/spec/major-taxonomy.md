# AI Agent 专业分类白皮书

**版本**: v0.1-alpha
**日期**: 2025-03-11
**状态**: 草案讨论

---

## 📋 文档说明

本文档基于 2024-2025 年行业调研，为 AI Academy 制定专业的分类体系和标准规范。

---

## 🎯 分类设计原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **岗位导向** | 每个专业对应一个明确的职业岗位能力 |
| **能力完整** | 专业应提供岗位所需的全套能力，而非零散技能 |
| **标准化交付** | 统一的专业结构、元数据、部署流程 |
| **可扩展性** | 支持专业之间的依赖关系和能力组合 |
| **社区共建** | 基于开源方案积累学校特色能力 |

### 1.2 设计依据

基于以下行业调研成果：
- **Surge AI 能力金字塔**（2025）：4层能力体系
- **Digital Twin Consortium 能力周期表**（2025）：6核心领域
- **五级进阶模型**（2024）：L0-L5 自主性分级
- **认知行为框架**（2024）：感知-思考-执行-适应

## 🏫 专业分类体系

### 2.1 一级分类（按应用领域）

基于 AI Agent 在 2024 年的主流应用场景，划分 5 大一级分类：

| 分类代码 | 分类名称 | 覆盖场景 | 对应行业 |
|----------|----------|----------|----------|
| **OA** | **办公自动化** | 邮件管理、文档处理、日程管理 | 企业办公、个人助理 |
| **DA** | **数据分析** | 数据清洗、可视化、洞察生成 | 金融、电商、运营 |
| **DC** | **内容创作** | 文案生成、设计辅助、多媒体制作 | 媒体、营销、自媒体 |
| **CS** | **客户服务** | 客服问答、工单处理、售后支持 | 电商、服务业、SaaS |
| **DT** | **开发工具** | 代码生成、测试、CI/CD 自动化 | 软件开发、DevOps |

### 2.2 二级分类（按岗位能力）

每个一级分类下细分为具体的专业（二级分类）：

#### OA - 办公自动化

| 专业代码 | 专业名称 | 核心能力 | 对应岗位 |
|----------|----------|----------|----------|
| OA-EM | 邮件管理专业 | 邮件分类、智能回复、归档整理 | 邮件助理、行政专员 |
| OA-DOC | 文档处理专业 | 文档生成、格式转换、内容提取 | 文档专员、秘书 |
| OA-CAL | 日程管理专业 | 日程安排、会议协调、提醒通知 | 行政助理、秘书 |
| OA-NOTE | 知识管理专业 | 笔记整理、知识库构建、快速检索 | 知识专员、研究员 |

#### DA - 数据分析

| 专业代码 | 专业名称 | 核心能力 | 对应岗位 |
|----------|----------|----------|----------|
| DA-REP | 报表生成专业 | 数据汇总、自动报表、趋势分析 | 数据分析师、BI专员 |
| DA-INS | 洞察挖掘专业 | 异常检测、关联分析、预测建模 | 数据科学家 |
| DA-VIS | 可视化专业 | 图表生成、仪表盘搭建、交互展示 | 数据可视化师 |

#### DC - 内容创作

| 专业代码 | 专业名称 | 核心能力 | 对应岗位 |
|----------|----------|----------|----------|
| DC-TEXT | 文案写作专业 | 标题生成、内容创作、SEO优化 | 内容运营、文案策划 |
| DC-MULTI | 多媒体创作专业 | 图像生成、视频剪辑、音频处理 | 新媒体运营、视频创作者 |
| DC-SOCIAL | 社媒运营专业 | 脚本生成、发布自动化、互动管理 | 社媒运营、KOL |

#### CS - 客户服务

| 专业代码 | 专业名称 | 核心能力 | 对应岗位 |
|----------|----------|----------|----------|
| CS-QA | 智能问答专业 | FAQ检索、问题解答、知识库维护 | 在线客服 |
| CS-WO | 工单处理专业 | 工单分类、自动流转、进度跟踪 | 客服经理 |
| CS-CRM | 客户管理专业 | 客户画像、需求挖掘、精准营销 | 客户成功经理 |

#### DT - 开发工具

| 专业代码 | 专业名称 | 核心能力 | 对应岗位 |
|----------|----------|----------|----------|
| DT-CODE | 代码生成专业 | 需求转代码、代码补全、重构优化 | 软件工程师 |
| DT-TEST | 测试自动化专业 | 测试用例生成、自动执行、缺陷报告 | 测试工程师 |
| DT-OPS | DevOps 专业 | CI/CD配置、日志分析、故障排查 | DevOps工程师 |

## 📦 专业能力包标准

### 3.1 专业元数据规范

每个专业必须包含以下元数据：

```yaml
major:
  code: "OA-EM"                    # 专业代码（格式：一级分类-二级分类）
  name: "邮件管理专业"              # 专业名称
  description: "提供邮件分类、智能回复、归档整理等全套邮件管理能力"
  version: "1.0.0"                 # 版本号（语义化版本）
  category: "OA"                   # 所属一级分类
  author: "AI Academy"             # 作者
  created_at: "2025-03-11"         # 创建日期
  updated_at: "2025-03-11"         # 更新日期

  # 能力层级（参考 Digital Twin Consortium 能力周期表）
  capabilities:
    - domain: "perception"         # 感知/知识
      level: 3                     # 成熟度等级（1-5）
      skills:
        - "邮件内容理解"
        - "意图识别"
    - domain: "cognition"          # 认知/推理
      level: 3
      skills:
        - "优先级判断"
        - "分类逻辑"
    - domain: "action"             # 行动/执行
      level: 3
      skills:
        - "自动回复"
        - "归档整理"
    - domain: "adaptation"         # 学习/适应
      level: 2
      skills:
        - "用户偏好学习"

  # 技术依赖
  dependencies:
    - name: "email-api"
      version: ">=1.0.0"
      description: "邮箱API访问能力"
    - name: "nlp-skill"
      version: ">=2.0.0"
      description: "自然语言处理能力"

  # 前置配置
  prerequisites:
    - type: "email_oauth"
      required: true
      description: "邮箱授权（OAuth 2.0）"
      guide: "配置邮箱授权文档链接"
    - type: "folder_structure"
      required: false
      description: "自定义邮件文件夹结构"

  # 学习目标
  learning_outcomes:
    - "能够自动分类和优先级排序邮件"
    - "能够根据上下文生成智能回复"
    - "能够自动归档和清理过期邮件"

  # 质量标准
  quality_metrics:
    - metric: "分类准确率"
      threshold: ">=85%"
    - metric: "回复相关性"
      threshold: ">=90%"
    - metric: "处理速度"
      threshold: "<3秒/封"
```

### 3.2 专业目录结构规范

每个专业能力包必须遵循以下目录结构：

```
majors/{专业代码}/
├── major.yaml              # 专业元数据（必需）
├── skills/                 # 技能清单（必需）
│   ├── skill-list.yaml     # 技能列表
│   └── {skill-id}/         # 技能包目录
│       ├── skill.yaml      # 技能元数据
│       ├── prompt/         # Prompt 模板
│       ├── code/           # 代码实现
│       └── config/         # 配置文件
├── prompts/                # 系统级 Prompt（必需）
│   ├── system.md           # 系统提示词
│   ├── examples.md         # 示例对话
│   └── guidelines.md       # 使用指南
├── mcp/                    # MCP 组件（可选）
│   └── {mcp-id}/
│       ├── config.yaml     # MCP 配置
│       └── schema/         # 接口定义
├── examples/               # 使用示例（必需）
│   ├── basic/              # 基础示例
│   └── advanced/           # 高级示例
├── tests/                  # 测试用例（必需）
│   ├── unit/               # 单元测试
│   └── integration/        # 集成测试
└── docs/                   # 文档（必需）
    ├── quickstart.md       # 快速开始
    ├── api.md              # API 文档
    └── troubleshooting.md  # 故障排查
```

### 3.3 技能清单规范

```yaml
skills:
  - id: "skill-001"
    name: "邮件分类"
    type: "skill"            # skill | prompt | mcp
    version: "1.0.0"
    description: "根据邮件内容、发件人、时间等维度自动分类"
    category: "perception"   # perception | cognition | action | adaptation
    maturity_level: 3        # 1-5
    capabilities:
      - "多维度分类"
      - "自定义规则"
    source: "open-source"    # open-source | custom | third-party
    source_url: "https://github.com/xxx/xxx"
    integration:
      type: "api"
      endpoint: "/api/email/classify"
    quality:
      accuracy: ">=85%"
      latency: "<500ms"
```

## 🎓 专业能力分级标准

### 4.1 基于 Digital Twin Consortium 的 5 级成熟度

| 等级 | 名称 | 能力描述 | 适用场景 |
|------|------|----------|----------|
| **Level 1** | 基础自动化 | 规则驱动，固定流程 | 简单任务（如自动回复固定话术） |
| **Level 2** | 自适应学习 | 从反馈中学习优化 | 半结构化任务（如邮件分类） |
| **Level 3** | 上下文感知 | 理解复杂上下文 | 结构化任务（如会议总结） |
| **Level 4** | 协作智能 | 多 Agent 协同 | 复杂任务（如项目协作） |
| **Level 5** | 创造性生成 | 超越人类的创新 | 高创造性任务（如创意设计） |

### 4.2 能力域划分

| 能力域 | 说明 | 典型能力 |
|--------|------|----------|
| **Perception** | 感知与知识获取 | 文本理解、图像识别、数据提取 |
| **Cognition** | 认知与推理 | 逻辑推理、决策判断、优先级排序 |
| **Action** | 行动与执行 | 工具调用、API调用、任务执行 |
| **Interaction** | 交互与协作 | 对话管理、用户引导、多Agent协作 |
| **Learning** | 学习与适应 | 用户偏好学习、反馈优化、知识更新 |
| **Governance** | 治理与安全 | 权限控制、合规检查、风险管控 |

## 🔄 专业依赖与组合

### 5.1 依赖关系规范

专业之间可以声明依赖关系：

```yaml
dependencies:
  majors:
    - code: "OA-EM"
      version: ">=1.0.0"
      reason: "邮件管理专业依赖知识管理专业的能力"
  skills:
    - id: "skill-nlp"
      version: ">=2.0.0"
```

### 5.2 推荐组合

| 组合名称 | 包含专业 | 适用场景 |
|----------|----------|----------|
| **办公全能包** | OA-EM + OA-DOC + OA-CAL + OA-NOTE | 企业办公助理 |
| **内容创作包** | DC-TEXT + DC-MULTI + DC-SOCIAL | 新媒体运营 |
| **数据洞察包** | DA-REP + DA-INS + DA-VIS | 数据分析师 |
| **客户成功包** | CS-QA + CS-WO + CS-CRM | 客服团队 |
| **开发效能包** | DT-CODE + DT-TEST + DT-OPS | 软件开发团队 |

## 📊 质量评估标准

### 6.1 专业质量维度

| 维度 | 指标 | 标准 |
|------|------|------|
| **功能性** | 能力覆盖率 | >=90% 岗位核心能力 |
| **准确性** | 任务成功率 | >=85% |
| **效率** | 响应时间 | <3秒 |
| **稳定性** | 错误率 | <5% |
| **可维护性** | 代码规范度 | >=90% 符合规范 |
| **文档完整性** | 文档覆盖率 | 100% 必需文档 |

### 6.2 技能质量评估

每个技能必须通过以下测试：

```yaml

    latency: "<1s"
```
